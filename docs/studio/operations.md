# Fasihi Standard Operating Procedures

This document outlines how Fasihi operates as an asynchronous, distributed-first software engineering studio in Kenya.

## Communication Principles
* **Writing Over Talking:** If a design decision or technical change can be written down, it must be documented in a Notion RFC (Request for Comments) rather than discussed in a meeting.
* **No Meeting Wednesdays:** Wednesdays are dedicated to deep work. No internal meetings may be scheduled.
* **Public Channels First:** Avoid private direct messages for project decisions. Ask questions in public Slack channels so the AI Company Brain can index them.

## Pull Request Guidelines
* All code changes must be submitted via GitHub Pull Requests (PRs).
* Every PR requires a review from at least one peer engineer (e.g., James for frontend, Sarah for backend storage, Amina for agent integration).
* Main branch deployments are triggered automatically via GitHub Actions pipelines.
